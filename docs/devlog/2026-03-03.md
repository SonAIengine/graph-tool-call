# 2026-03-03 — 방향 전환: 스펙 품질은 소스에서 해결

## 배경

graph-tool-call v0.3.0을 X2BEE BO API (1,077 endpoints)에 적용.
4가지 이슈를 발견했으나, 분석 결과 절반은 **스펙 품질 문제**임을 확인.

## 핵심 판단

> Edge 폭발과 requestBody 누락은 retrieval engine 버그가 아니라 OpenAPI 스펙 자체의 문제.
> graph-tool-call에 workaround를 패치하는 것은 잘못된 방향.
> 스펙 품질을 소스에서 해결하는 **별도 lint 도구**가 올바른 접근.

## Issue 1: Edge 폭발 → ai-api-lint

- 현상: 경로 유사도 기반 edge가 폭발적으로 생성
- 원인: 1,077 endpoints 대부분이 `/bo/` prefix를 공유 → 경로 유사도가 과도하게 높음
- 판단: 이는 스펙의 path design 문제 (너무 flat한 구조, 의미 없는 공통 prefix)
- 해결: ai-api-lint의 path_design 규칙 (AI050~AI053)으로 스펙 단계에서 경고

## Issue 2: requestBody 누락 → ai-api-lint

- 현상: POST endpoint에 requestBody가 없어 도구 파라미터 정보 부재
- 원인: 스펙 작성 시 requestBody를 생략한 것 (코드 자동생성 한계)
- 판단: retrieval engine이 없는 정보를 만들 수는 없음
- 해결: ai-api-lint의 request_body 규칙 (AI030~AI034)으로 스펙 단계에서 에러

## Issue 3: 한글 BM25 → graph-tool-call 개선

- 현상: "정기주문해지" 같은 한글 합성어가 단일 토큰으로 처리되어 검색 실패
- 원인: `_tokenize()`가 ASCII camelCase만 분리, 한글은 공백 단위만 분리
- 해결: 한글 character bigram 토큰화 추가 (외부 의존성 없이 re만 사용)
- 예시: "정기주문해지" → ["정기주문해지", "정기", "기주", "주문", "문해", "해지"]

## Issue 4: Malformed Parameter → graph-tool-call 테스트

- 현상: name 필드 없는 parameter에서 KeyError
- 원인: OpenAPI 스펙의 parameter에 필수 필드(name) 누락
- 해결: 코드는 이미 수정 완료 (`if "name" not in p: continue`), 테스트만 추가

## 새 프로젝트: ai-api-lint

- 위치: `~/Projects/personal/ai-api-lint`
- 목적: OpenAPI 스펙의 AI/LLM agent 친화성을 0-100으로 평가
- 7개 카테고리, 31개 규칙: operationId, description, parameters, request_body, response, path_design, mcp_readiness
- zero-dependency 코어 (graph-tool-call 철학 계승)
- graph-tool-call과의 관계: 전처리 도구 (lint → fix → ingest)

## 논문 관점

이 경험은 논문의 핵심 주장을 강화:
- 도구 검색 성능은 **스펙 품질에 크게 의존**
- Graph-based retrieval의 한계가 아닌 입력 품질의 한계
- ai-api-lint → graph-tool-call 파이프라인이 실용적 contribution
